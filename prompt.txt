The optimization strategy has changed, so run_scenarios_loop is no longer necessary and has been removed. Here is the new strategy. Perf is no longer needed; its logic can be deleted.
1. Hook all the steps that involve different configurations together using a bash script, instead of a new Rust file (like paxos_optimizations).
2. First, we will run a network calibration pass. Create an example for network_calibrator that takes in message size in bytes as a parameter. The bash script will launch multiple of these in parallel with exponentially increasing message sizes, up to 4KB. It will then parse the resulting SAR metrics at the measurement second for each configuration of clients, divide by throughput to get cost-per-byte, and store the results in a file. This only needs to be run once across all benchmarks.
3. We then benchmark the protocol baseline. Run benchmark_paxos (without counters) until completion. Do this by adding a flag to deploy_and_analyze (and run_benchmark).
4. We can concurrently decouple everything that can be decoupled in the protocol and benchmark each operator's usages separately. Add a flag to benchmark_paxos so if true, it runs greedy_decouple_analysis with SAR metrics before deploying. Modify greedy_decouple_analysis so it applies to all locations except excluded ones.
It then gathers (at measurement_second) the SAR metrics for each decoupled node. It multiplies cost-per-byte of network (calibrated earlier) by the number of network bytes, and subtracts that cost from the measured cost, getting a cost-per-operator. This cost is then associated with the 1st operator (out of all operators assigned to this machine, since we know they cannot be decoupled further anyway). The cardinality (from the counter) is also associated with each operator.
5. To figure out input/output byte sizes for each operator, create a function in deploy_and_analyze similar to insert_count, that takes the output of greedy_decouple_analysis, and wherever a network would be inserted, it instead inserts an Inspect that serializes every Nth input and outputs the size of the tuple, which is then collected by a new metric in MetricLogs. Also insert this after any pre-existing Network.
Runs this over benchmarkPaxos (in concurrently with everything else), compute the median sizes for each operator, associate that bytes-per-tuple with each operator.
5. When all the previous concurrent steps complete, combine this data. Create a HashMap of op_to_output_bytes, op_to_count, and op_to_cost (where cost includes a separate field for CPU, I/O, and memory).
6. Find the current bottleneck through the RuntimeMetadata function.
7. Apply decouple + partition analysis to the bottleneck by running the ILP and allowing the usage of 2, 3, etc machines until the min resource usage across nodes no longer decreases. Resource usage is calculated by summing the cost-per-operator of all operators assigned to that machine, and then the cardinality * bytes-per-tuple * cost-per-byte for all existing (and newly created) networks. Very similar logic is already in place, except the only cost considered right now is CPU and network doesn't account for bytes.
Modify the logic for partition_ilp_analysis to account for the number of machines; if there are 2 machines allotted, then we can either decouple or partition the whole thing and halve cost. If there are 3 machines allotted, then we can also decouple and partition 1 of them. Etc.
Store the decisions for each configuration on a file.
8. Apply the decision with the fewest number of machines, run the benchmark again, and run analysis again if throughput improves. If throughput does not improve, or analysis on the bottleneck does not find a benefit to having > 1 machines, then halt.
